What's new

RDS Bridge

RDS Bridge is a browser-based RDS decoder built for the thing ordinary decoders struggle with: pulling station identification out of weak, fading FM signals — the kind you chase when DXing. It works two ways: live, paired with SDRConnect for real-time decoding and tuning; or offline, decoding recorded IQ .wav files from any SDR — SDRuno, SDR Console, SDR#, HDSDR and the rest — with no SDRConnect needed.

Runs in a Chromium browser — Chrome, Edge, Brave or another Chromium-based browser. Safari and Firefox aren't supported: RDS Bridge relies on browser audio and streaming features they don't fully implement. Full details, guides and downloads are at rdsbridge.com.

Getting started — live, with SDRConnect

  • In SDRConnect, enable its remote/WebSocket server, and tune your FM station as usual.
  • In RDS Bridge, with the source set to SDRConnect: Connect → choose your device → Start. The default address ws://127.0.0.1:5454 is for SDRConnect running on the same machine.
  • Within a second or two of a lock you'll see PI, the station name, and the rest fill in.

Getting started — offline, from an IQ recording

  • Record a 2-channel I/Q .wav across the FM band in your SDR software (SDRuno, SDR Console, SDR#, HDSDR …). A wider capture lets you tune across more of the band; you need at least 120 kHz of sample rate for the 57 kHz RDS subcarrier to be present at all.
  • In RDS Bridge, set the source to IQ File and drop the recording onto the panel (or click to choose). It feeds the same decoder — no SDRConnect needed.
  • If the recording carries its centre frequency — in the file's metadata (SDR Console) or its filename (SDRuno, SDRConnect) — tuning is absolute: type a station or click the RF waterfall and the readout shows real MHz. Without it the file still decodes, at its own centre. Use the transport to play, pause and scrub anywhere in the recording; both waterfalls and the audio follow.

Tuning and band scan in MPX mode

  • MPX decodes the composite your SDR software sends, so tuning normally stays in that software. If you run the helper against a radio it can control over CAT (for example SDR Console), RDS Bridge can tune it for you: the frequency readout above becomes editable — type a frequency and press Enter, exactly as in the other source modes — and the band scan becomes available.
  • With any other MPX source the readout stays read-only and the scan is unavailable, because there is nothing for Bridge to tune. Tune in your SDR software instead.
  • The MPX band scan is slow, and that is unavoidable. Bridge retunes your radio one channel at a time and must wait roughly two seconds on each for the audio to travel through the demodulator and virtual audio cable and lock. There is no wideband spectrum in MPX, so empty channels cannot be skipped the way they are on a live SDRConnect stream — every channel is given the full look. A complete band pass takes several minutes.
  • How to use it well: for routine monitoring build a watch list of the channels you actually care about — only those are visited, so a pass takes seconds. For band-watching leave DX watch looping: it learns which channels are dead and skips them on later passes, so it speeds up as it goes. Save the full-band sweep for when you want a complete picture, such as during a lift.
  • Decoding is unaffected by the speed: a station that appears is decoded exactly as it would be on any other source, through the same PI commit guard, so nothing is fabricated.

What you're looking at

  • PI — the station's unique ID code (in hex). It's the hardiest piece of data and locks first, often before anything else is readable.
  • PS — the 8-character station name. It needs more signal than PI, so on a weak catch you may see PI long before the name finishes assembling.
  • PTY — programme type. RadioText — the longer scrolling message some stations send.
  • TP / TA — traffic-programme and traffic-announcement flags. Stereo — the 19 kHz pilot is present.
  • AF — alternative frequencies the station lists. Clock — time and date, if broadcast.

The band, tuning & the 57k panel

  • RF waterfall — the captured band, with an MHz scale and a green marker on the channel being decoded. Click-to-tune: click the waterfall (or its scale) to retune SDRConnect's VFO there; snap rounds the click to a frequency step (default 100 kHz) to match your tuning grid.
  • 57k confidence — an advisory estimate of how likely this pass is to yield an ID. It accumulates evidence across an opening rather than reacting to each fade, so it stays meaningful through rapid QSB; it reads "PI captured" / "name captured" once a hard ID is confirmed. It's an estimate, not a calibrated probability.
  • 57k constellation — the post-carrier symbol samples plotted live. A diffuse cloud is noise; two tight poles forming on the horizontal axis means coherent, recoverable RDS. Watch it crystallise as a station comes up — it shows how close to decodable a signal is in a way the spectrum can't.

Lock & quality — what matters for DX

  • Sync — whether the decoder has block alignment. "lock" is good; "hunting" means it's still searching.
  • Pilot — the 19 kHz stereo pilot. A strong, steady pilot means there's a genuinely coherent signal to work with; a weak one usually means there simply isn't enough signal to decode, however hard you try.
  • Data Q — how clean the RDS bitstream is (the share of blocks arriving intact).
  • SNR — signal-to-noise, reported by SDRConnect.

Two knobs worth trying on stubborn signals

  • Channel bandwidth — a slider (or type-in kHz field, 120–230 kHz) around the tuned station. Narrowing it rejects splatter from adjacent channels and can rescue a weak station crowded by louder neighbours; widen it for clean, strong signals. Worth knowing how this differs from a normal radio: most SDRs have one filter that sets the audio, the display and what's decoded all at once. RDS Bridge decodes from the full-bandwidth IQ with its own filter, so this control moves two filters kept in step — RDS Bridge's decode filter (always), and, when SDRConnect allows hardware control, SDRConnect's front-end filter too (so you also hear it in the audio and see it move on SDRConnect's own display). The cyan edges on the RF waterfall show the current passband, and track live as you drag. If SDRConnect reports no hardware control, only the decode filter moves — silent, but still working.
  • Error correction — how hard the decoder works to repair damaged blocks. ≤2 bits is a sound default; ≤3 recovers more at a slightly higher chance of a wrong fix; off gives detect-only, the purest reads.

Tips

  • If PI flickers in but the name won't complete, the signal's marginal — narrow the bandwidth and give it time; the name rebuilds character-by-character from repeated transmissions.
  • A solid pilot but no decode usually points to interference rather than weak signal — narrowing bandwidth helps more than correction there.
Playing back IQ files is a normal, everyday feature (the IQ File source above) — you don't need advanced mode for it. Advanced mode (top-right, or add ?dev=1 to the address) adds developer tools on top of that: calibrated noise/fading injection and the decode-threshold auto-sweep for IQ files, experimental decode read-outs, and low-level DSP switches. You don't need any of it for normal listening. One of those read-outs, match, is a hint rather than a decode: it watches for stations you've recently caught on this frequency and flags when the signal looks consistent with one of them. It never changes the decoded PI or name — treat it as a nudge worth a second look, not an identification.

RDS Bridge decodes RDS entirely in your browser, in a background worker, from the raw I/Q stream SDRConnect sends over WebSocket. What sets it apart from a textbook decoder is its acquisition front-end, designed to lock signals far too weak or unstable for a conventional loop-based decoder to catch from cold.

RDS, briefly

The RDS subcarrier sits at 57 kHz — the third harmonic of the 19 kHz stereo pilot — carrying 1187.5 bits/s. The data is differentially encoded and bi-phase (Manchester-style) modulated onto a suppressed-carrier DBPSK signal. Bits are grouped into blocks of 26 (16 information + 10 check), four blocks to a group. Each block type carries a distinct "offset word" that both identifies it and drives synchronisation. The PI code occupies block A of every group, so it repeats about 11 times a second — which is why it's the most recoverable field.

The signal chain

I/Q in → front-end numerically-controlled mixer shifts the wanted station to baseband and a polyphase filter decimates to a manageable processing rate → FM discrimination to the multiplex → band-pass around 57 kHz → DBPSK carrier recovery → bi-phase symbol recovery → differential decode → block synchronisation → group parsing into the fields above.

The headline: NDA open-loop acquisition

A conventional decoder closes three feedback loops at once from a cold start — a Costas loop for the DBPSK carrier, a timing loop (e.g. Gardner) for symbol timing, and a PLL for the pilot. On weak or fading signals these loops fight each other: timing can't settle until the carrier is roughly right, and the carrier can't settle until timing is roughly right. The result is a hang — the decoder never converges even though a perfectly decodable signal is present. This was the single biggest cause of missed weak catches.

RDS Bridge instead acquires open-loop and non-data-aided (NDA). It estimates symbol timing feed-forward with an Oerder–Meyr-style estimator that recovers the symbol-rate spectral line directly from the signal — no loop, nothing to hang — and uses that to seed the Costas carrier phase, while holding the Gardner timing feedback disabled. Only once the front-end has a coarse, confident lock does it hand over to the closed loops for fine tracking. The effect is that it converges from cold, quickly and repeatably, on signals where loop-based decoders stall. This is the core advance of the project and the validated reason it outperforms the reference decoder on weak FM DX.

Symbol recovery, matched filtering, gear-shift

The bi-phase symbols are matched-filtered (a Manchester-matched response by default) before timing recovery, maximising SNR at the decision instant. A "gear-shift" strategy runs the tracking loops wide during pull-in and narrows them once locked, trading acquisition speed for steady-state noise rejection.

Differential decode and the 180° ambiguity

Because the carrier is suppressed, its recovered phase has an inherent 180° ambiguity. RDS sidesteps this by differential encoding — each data bit is the relationship between consecutive channel symbols — so a global phase inversion leaves the decoded data unchanged.

Block sync and error correction

Synchronisation is found by sliding the 26-bit window and testing its syndrome against the five known offset words; a run of consistent offsets declares lock. The (26,16) shortened cyclic code can repair burst errors, and the decoder applies syndrome-based correction up to a configurable maximum burst length (the Error-correction control) — letting you trade recovery against the small chance of a mis-correction.

PI repetition guard

Block A carries the PI but, beyond its offset-word syndrome, has no message-level checksum — so a single burst-corrected read can pass as a valid-looking but wrong PI, especially on a fading signal. The decoder therefore requires the same PI to be received more than once (default: one confirming repeat) before it is committed to the display, the DX log, or the confidence panel. Tunable in advanced mode. A genuine PI repeats constantly; a spurious correction rarely lands twice on the same value.

57k confidence & constellation

The confidence read is deliberately not an instantaneous reading — under Es-style rapid fading those are meaningless. Instead it ratchets accumulated evidence over the opening (peak SNR and pilot, pilot/sync duty cycles, soft-PI/blind convergence, groups parsed), rising quickly on a surge and bleeding slowly, and banks a sticky floor when a hard PI or name is confirmed. It reads existing telemetry only and is an estimate, not a calibrated probability. The constellation plots the post-carrier symbol samples directly: coherent RDS collapses to two poles on the decision axis, noise stays a cloud — the genuine discriminator between a real subcarrier and energy that merely sits at 57 kHz. Both are pure observers of the decode path and never feed back into it.

Experimental paths (advanced, off by default)

Soft-PI integration keeps a confidence-weighted running tally of each PI bit across all groups — including sub-threshold blocks a normal decoder discards — so faint-but-consistent evidence accumulates while noise averages out, in principle reading the ID from signals too weak to decode block-by-block. Blind soft-PI extends this to work without ever achieving hard sync, by running a bank of 104 candidate group-phase accumulators in parallel and letting the true phase self-validate against the offset-A syndrome.

Honest result: in controlled testing (the harness below), both proved at best marginal over the NDA baseline on real signals — under steady noise they tie the normal decoder and don't meaningfully extend the threshold. They're retained, off by default, as the foundation for future work such as cross-referencing a signal against a previously-decoded PI on the same frequency. The frozen baseline is the NDA decoder alone.

Known-PI matcher (advanced, advisory)

The matcher flips the question from "what PI is this?" to "is a station I already know present?". From the DX log it takes the PIs caught in the last half-hour on this same frequency, and for each one it correlates the live soft accumulator against that station's known 26-bit block A — all 26 bits (information and check), since once the PI is assumed the whole block is determined. Because it scores a known pattern rather than decoding unknown bits, a single marginal bit contributes proportionally instead of breaking a syndrome — so in principle it can confirm a familiar station a little deeper into the noise than a cold decode manages. That real-world gain is not yet measured on live signals, so the matcher sits in the same experimental tier as the soft/blind read-outs: treat it as a hint, not a result, until the threshold sweep quantifies it. The match read-out shows the candidate, its remembered name and the percentage agreement; it turns solid only when agreement clears an absolute floor and beats the next-best candidate by a margin, so a busy band can't coax a false confirm.

It is strictly advisory. The matcher only ever reads the DX log and only ever annotates — it never writes the log, never changes the decoded PI, and cannot produce a PS (it has no name evidence; the name beside a match is simply what was logged for that PI earlier). The decoder's own PI and PS, from genuine block decodes, remain the only identification. Read it as a second opinion — "this is consistent with Tyneside, caught 4 minutes ago here" — and you remain the judge.

The test harness (advanced)

Drops a recorded I/Q .wav straight into the worker pipeline — no SDRConnect needed — and can inject calibrated noise and fading to measure exactly where each method loses lock. Its auto-sweep steps the SNR down and reports the weakest level each decode path survives, turning "feels better" into a figure in dB.
